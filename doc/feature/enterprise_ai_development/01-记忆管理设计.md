## 项目背景

**项目名称**：pet-care-ai
**技术栈**：Spring AI + 智谱AI + Elasticsearch + Redis
**当前架构**：单步 Function Calling，Redis 存储会话级短期记忆
**本文档目标**：实现长期语义记忆，支持跨会话知识复用

---

## 一、现状分析

### 1.1 已有能力
- Redis 存储会话级短期记忆（MessageWindowChatMemory，maxMessages=10）
- ElasticsearchClient + EmbeddingModel 基础设施完备
- 向量维度 1024，cosine 相似度

### 1.2 存在的问题
| 问题 | 描述 |
|------|------|
| 会话后历史丢失 | 会话结束后无法复用 |
| 重复提问体验差 | 用户重复问相同问题 |
| 无法个性化推荐 | 不知道用户偏好、预算等 |
| 数据价值浪费 | 历史对话无法分析挖掘 |

## 二、架构设计

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                      用户请求                             │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  SemanticMemoryAdvisor（新增）                           │
│  1. 生成 query 向量                                       │
│  2. ES 语义检索相关历史（top 3）                          │
│  3. 拼接到 Prompt 上下文                                  │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  PromptChatMemoryAdvisor（已有）                          │
│  从 Redis 获取会话窗口（最近10条）                        │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  ChatClient + Tool Calling                               │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  ChatMessageSink（新增）                                 │
│  异步存储对话到 ES                                        │
└─────────────────────────────────────────────────────────┘
```

### 2.2 数据流

**写入流**（对话完成后异步执行）：
```
ChatClient 响应 → ChatMessageSink → 生成向量 → ES 存储
```

**读取流**（每次请求前）：
```
用户 query → 生成向量 → ES KNN 检索 → 拼接到 Prompt
```

## 三、数据结构设计

### 3.1 索引定义

**索引名**：`chat_history`

**Mapping**：
```json
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1
  },
  "mappings": {
    "properties": {
      "id": {"type": "long"},
      "conversation_id": {"type": "keyword"},
      "user_id": {"type": "long"},
      "session_id": {"type": "keyword"},
      "role": {"type": "keyword"},
      "content": {"type": "text", "analyzer": "ik_max_word"},
      "embedding": {
        "type": "dense_vector",
        "dims": 1024,
        "index": true,
        "similarity": "cosine"
      },
      "metadata": {
        "type": "object",
        "properties": {
          "tool_calls": {"type": "keyword"},
          "tokens_used": {"type": "integer"},
          "model": {"type": "keyword"}
        }
      },
      "created_at": {"type": "date"},
      "expires_at": {"type": "date"}
    }
  }
}
```

### 3.2 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| conversation_id | keyword | 对话ID（多轮会话共享） |
| user_id | long | 用户ID |
| session_id | keyword | 会话ID（一次访问） |
| role | keyword | user / assistant / system |
| content | text | 消息内容 |
| embedding | dense_vector | 向量（仅 user 消息需要） |
| metadata | object | 扩展信息（工具调用、token消耗等） |
| expires_at | date | 过期时间（隐私保护） |

## 四、接口定义

### 4.1 SemanticMemoryAdvisor

**职责**：在 LLM 调用前检索相关历史

**输入**：
- 用户 query
- user_id
- conversation_id

**输出**：
- 格式化的历史上下文（拼接为 Prompt）

**检索策略**：
1. 生成 query 向量
2. KNN 检索：user_id 过滤 + cosine 相似度
3. top_k = 3，min_score = 0.75
4. 按时间倒序排列

### 4.2 ChatMessageSink

**职责**：对话完成后异步存储

**触发时机**：
- ChatClient 返回响应后
- 使用 `@EventListener` 或 AOP

**存储内容**：
- 用户消息 + 向量
- AI 响应（不存向量）
- 元数据（工具调用、token）

## 五、边界问题处理

### 5.1 隐私保护

| 问题 | 方案 |
|------|------|
| 敏感信息泄露 | expires_at 字段，默认 90 天过期 |
| 数据删除 | 提供"清除历史"API，物理删除 |
| 检索权限 | 必须 user_id 过滤，防止跨用户 |

### 5.2 性能优化

| 问题 | 方案 |
|------|------|
| 向量生成延迟 | 异步存储，不阻塞主流程 |
| ES 检索延迟 | 设置超时 500ms，失败降级 |
| Redis 压力 | 历史检索不走 Redis，保持现有逻辑 |

### 5.3 存储策略

| 场景 | 策略 |
|------|------|
| 仅 user 消息存向量 | assistant 消息不参与检索 |
| system 消息不存 | 无需检索的内部消息 |
| 重复内容去重 | 同一 conversation_id + content 去重 |

## 七、风险点

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| ES 检索超时 | 用户体验下降 | 降级处理，超时则跳过语义检索 |
| 向量生成成本 | Token 消耗增加 | 仅 user 消息生成向量 |
| 存储容量增长 | ES 磁盘压力 | 设置 90 天过期，定期清理 |
| 检索噪音 | 拼接无关历史 | 提高相似度阈值，top_k=3 |

### 9.2 API 接口

#### 9.2.1 清除历史记录（P1 必须实现）

**请求**
```
DELETE /ai/chat/history
Authorization: Bearer {accessToken}
```

**响应**
```json
{
  "success": true,
  "deletedCount": 150
}
```

**业务逻辑**
- 从 `UserContext` 获取当前用户 ID
- 删除 ES 中 `chat_history` 索引下该用户的所有数据
- 删除 Redis 中该用户的会话记忆（MessageWindowChatMemory）
- 返回删除数量

**后端实现要点**
```java
@DeleteMapping("/ai/chat/history")
public ResponseEntity<ClearHistoryResponse> clearHistory() {
    Long userId = UserContext.getUserId();
    // 1. 删除 ES 中的历史记录
    long deletedCount = chatHistoryRepository.deleteByUserId(userId);
    // 2. 清除 Redis 中的短期记忆
    chatMemoryRepository.deleteByUserId(userId);
    return ResponseEntity.ok(new ClearHistoryResponse(true, deletedCount));
}
```

#### 9.2.2 查询历史记录（P2 可选）

**请求**
```
GET /ai/chat/history?page=0&size=20
Authorization: Bearer {token}
```

**响应**
```json
{
  "total": 150,

    {
      "conversationId": "conv-123",
      "content": "我家金毛最近不爱吃饭",
      "createdAt": "2026-03-01T10:00:00"
    }
  ]
}
```
