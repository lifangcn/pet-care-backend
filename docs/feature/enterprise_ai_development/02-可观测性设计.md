## 项目背景

**项目名称**：pet-care-ai
**技术栈**：Spring AI + 智谱AI + Elasticsearch + Redis
**当前架构**：单步 Function Calling，仅 Console 日志
**本文档目标**：建立完整链路追踪，支持问题排查和性能分析

---

## 一、现状分析

### 1.1 已有能力

| 组件 | 用途 | 问题 |
|------|------|------|
| MyLoggerAdvisor | 打印请求/响应 | 仅文本日志，无结构化 |
| VectorStoreLogAspect | 记录 RAG 检索 | 无链路关联 |
| SimpleLoggerAdvisor | Spring AI 自带日志 | 颗粒度太粗 |
| Console Log | 应用日志 | 无法检索分析 |

### 1.2 存在的问题

| 问题 | 描述 | 影响 |
|------|------|------|
| 无链路追踪 | 无法追踪一次请求的完整路径 | 问题排查困难 |
| 无结构化日志 | ES 无法检索分析 | 数据价值浪费 |
| 无指标监控 | 不知道延迟、错误率、Token消耗 | 无法发现性能瓶颈 |
| 无异常告警 | 问题只能用户反馈 | 被动响应 |

## 二、架构设计

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                      ChatController                         │
│                     (conversationId)                         │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                   ObservabilityAspect                       │
│  1. 生成 traceId                                           │
│  2. 创建 ObservationContext                                │
│  3. 记录请求开始时间                                        │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    ChatClient + Advisors                    │
│  ┌─────────────────┐  ┌─────────────────┐                  │
│  │  RAG Retrieval  │  │  Tool Calling   │                  │
│  │  (记录检索结果)  │  │  (记录调用详情)  │                  │
│  └─────────────────┘  └─────────────────┘                  │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                  StructuredLogAppender                      │
│  1. 收集链路数据                                            │
│  2. 写入 ES（chat_trace 索引）                              │
│  3. 异步写入，不阻塞响应                                    │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    Kibana Dashboard                         │
│  - 链路追踪                                                  │
│  - 指标监控                                                  │
│  - 异常告警                                                  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 数据流

**同步流**（请求处理）：
```
请求 → ObservabilityAspect → 收集数据 → 放入 ThreadLocal
```

**异步流**（日志写入）：
```
响应完成 → 从 ThreadLocal 收集 → 结构化 → ES 异步写入
```

        }

      "tool_calls": {
        "type": "nested",
        "properties": {
          "tool_name": {"type": "keyword"},
          "arguments": {"type": "text"},

### 3.2 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| trace_id | keyword | 链路唯一标识（UUID） |
| conversation_id | keyword | 对话ID |
| user_id | long | 用户ID |
| duration_ms | integer | 总耗时（毫秒） |
| rag | object | RAG 检索详情 |
| tool_calls | nested | 工具调用列表（可能多个） |
| error | object | 错误信息（如有） |

## 四、接口定义

### 4.1 ObservabilityAspect

**职责**：拦截请求，创建追踪上下文

**拦截点**：`ChatController.ragChat()`

**处理流程**：
1. 生成 traceId（UUID）
2. 创建 ObservationContext，放入 ThreadLocal
3. 记录请求开始时间
4. 在响应完成后触发日志写入

### 4.2 ObservationContext

**职责**：存储一次请求的完整上下文

**数据结构**：
```
- traceId: String
- conversationId: String
- userId: Long
- startTime: Long
- request: RequestInfo
- response: ResponseInfo
- ragRetrieval: RAGInfo
- toolCalls: List<ToolCallInfo>
- error: ErrorInfo
- metadata: Metadata
```

### 4.3 StructuredLogAppender

**职责**：异步写入结构化日志到 ES

**触发时机**：
- 请求正常完成
- 请求异常失败

**写入内容**：
- 完整的 ObservationContext
- 序列化为 JSON

## 八、边界问题处理

### 8.1 性能影响

| 问题 | 方案 |
|------|------|
| ES 写入延迟 | 异步写入，使用独立的线程池 |
| 内存占用 | Context 大小限制，超过截断 |
| 日志量过大 | 设置采样率（如只记录 10%） |

### 8.2 数据隐私

| 问题 | 方案 |
|------|------|
| 敏感信息 | 用户消息可脱敏（可选） |
| 数据保留 | ILM 策略，30 天后自动删除 |
| 查询权限 | Kibana 权限控制 |

### 8.3 降级策略

| 场景 | 处理 |
|------|------|
| ES 不可用 | 降级为本地文件日志 |
| 写入失败 | 记录失败计数，不影响主流程 |
| ThreadLocal 泄漏 | 使用 try-finally 确保清理 |

## 九、风险点

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| ES 写入压力 | 影响主流程 | 异步写入 + 降级 |
| 日志量爆炸 | 存储成本高 | ILM 自动删除 + 采样 |
| ThreadLocal 泄漏 | 内存泄露 | 严格清理机制 |
| 链路断开 | 无法追踪 | 依赖系统透传 Header |
